Captain Scheduling Optimization for Eurowings

RWTH Aachen's Operations Research Practice Project with Eurowings (Lufthansa Group), supervised by Prof. Dr. Marco Lübbecke, with operational data provided by Lufthansa Industry Solutions: an integrated solver that builds and assigns complete, EASA-compliant captain rosters in a single optimisation. The column-generation engine is complete and runs end-to-end; current work is scaling it to the full instance. Ongoing (SS 2026).

The problem

Airline crew scheduling is one of the classic hard problems in operations research. Every captain needs a legal, contiguous roster; every flight leg needs exactly one captain; and the assignment must respect EASA flight-duty regulations alongside per-captain preferences — off-claims, vacation, and base-level reserve quotas. The usual industrial shortcut is to generate anonymous pairings first and roster them onto captains second. We don't: the pricing subproblem constructs legal duties, pairings, rest, reserve shifts and days off directly from raw legs into one personalised roster per captain. Each master column is a fully EASA-compliant roster for one captain.

Architecture

  • Master problem: Set Partitioning restricted master (RMP) with leg-coverage and daily base-reserve constraints, following Kasirzadeh et al. (2017). Branch-and-price is the planned extension.
  • Pricing subproblem: label-setting resource-constrained shortest path on a per-captain time-space DAG, with EASA 7/14/28-day rolling duty and 28-day flight-time resources, overtime at duty close, and off-claim deviation at the sink.
  • Solver: SCIP via PySCIPOpt, with a custom Pricer plugin registering negative-reduced-cost columns after each LP solve. Column generation with custom pricers is what SCIP is built for — Gurobi is kept only as a comparison benchmark.

Three design decisions worth telling

  • Consecutive days off enforced by topology, not by a resource. The network emits home-base rest only as blocks of 2 or 3 days; since 2 and 3 additively generate every integer ≥ 2, any legal rest length is representable while a lone 1-day off is structurally impossible. No DP resource needed.
  • Non-separable penalties live at the sink. Off-claim deviation depends on a captain's total days off across the horizon, which breaks label dominance. We track it on labels but exclude it from dominance, charging the penalty once at the sink.
  • Nominal cost recovered from duality, not re-summed. The pricer returns reduced cost; true cost is recovered as c = c_red + Σπ + μ + Σλ across all three dual families, keeping column objectives consistent as duals move.

The scaling story: measure, then fix the thing that's actually slow

The engine converged on toy instances but hit a wall on a 25-captain, 1,797-leg slice: building one captain's DAG took ~8.3 s and produced ~37k arcs, and the DAG was rebuilt per captain, per iteration — roughly 200 seconds of pure graph construction per pricing round before the solver could even re-solve the LP.

  • Topology caching. A captain's DAG mixes two things that change on different clocks: the shape (decided by the flight schedule and EASA timing rules — identical in iteration 1 and 50) and the arc costs (fresh duals every round). The fix: build the captain-independent skeleton once per base, re-stamp only the dual-dependent costs each iteration, and turn hard absences into per-captain node masks instead of rebuilds. The ~200 s/round build wall disappeared.
  • Bounded, vectorised construction. Replaced an O(legs²) iterrows double loop with a per-airport, departure-time-sorted bisect index, and capped outstation-overnight fan-out to the earliest legal next departures. Measured on a 644-leg weekly instance: total arcs −72% (16.7k → 4.7k), outstation arcs −93%, build time under 0.1 s — with all tests green, so arc semantics are preserved.
  • Exact base-group pricing. Once duals are fixed, every captain of a base sees the same monetary arc costs; the captain-specific terms (availability dual, off-claim target) apply only at the sink and cannot change label dominance. So all absence-free captains share one pricing sweep, disaggregated per captain afterwards. Measured: 70 → 27 sweeps per round, ~2.8× faster — and provably exact, pinned by a regression test asserting the converged LP is identical to per-captain pricing.
  • Well-posed instances as a correctness requirement. Early experiment slices sampled legs and captains independently, so demand wildly exceeded crew capacity — column generation burned hundreds of rounds chasing legs that were physically uncoverable. Built a per-base weekly instance generator that prints a demand/capacity verdict at generation time, so a converged LP is a genuine schedule rather than a slack-penalty artefact.

The recurring lesson: profile before optimising, and fix the multiplier, not the constant. Caching topology beat any solver tuning because it removed a cost paid captains × iterations times; base-group pricing then attacked the per-captain factor itself. Parallelising first would have meant more machines burning time on work that shouldn't be repeated at all.

My role

Within a 7-person team I built the core engine — the RMP (master_problem.py), the time-space network builder (network.py), the label-setting pricer (pricing.py) and the SCIP pricer plugin — plus all four scaling levers above. I also own the data and tooling layer: the Python loader reconciling 12 heterogeneous CSVs (mixed delimiters, German date formats, dual time zones), a 17-page field-by-field data guide with ER diagrams for the team, and the GitHub Actions CI running ruff + pytest on every push.

Note: the underlying crew and schedule data is confidential to Lufthansa Industry Solutions, so the repository is private; the figures above are aggregate statistics cleared for sharing.
