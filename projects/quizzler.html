The problem

Real-time multiplayer is the hard part of any trivia game: two players must be fairly matched, see each other's scores update live, and end with a result that can't be tampered with or lost to a dropped connection.

Real-time multiplayer core

  • Atomic matchmaking. Players enter a queue and are paired by a single Postgres RPC that stores the question set on the match row — eliminating the classic double-match race condition by design rather than by client coordination.
  • Live score sync with graceful degradation. Scores broadcast over a Supabase Realtime channel (WebSockets), and every realtime listener has a polling fallback, so a flaky connection degrades instead of breaking the match.
  • Authoritative settlement. A submit_score RPC settles the winner, updates ratings, and awards XP in one server-side transaction — no tampering, no partial writes.

Skill rating & progression

Each topic carries an independent ELO rating (K=32), so strength in history is tracked separately from sports. Scoring rewards both accuracy and speed (100 points plus a time bonus), and an XP system drives progression. Keeping the rating math server-side keeps it authoritative — the same transactional-integrity instinct that matters in any production data system.

Data & content

A normalized Postgres schema with row-level security backs anonymous (username-only) auth. Content comes from both an external trivia API and a built-in quiz builder for user-generated questions. Every release is gated by a verification step: TypeScript compile plus web export must pass before anything ships.

Why it belongs next to the ML and OR work

The threads connect: ELO is an iteratively-updated skill estimate under uncertainty, the scoring system is incentive design, and every match emits structured outcome data ready for fairness, retention, or difficulty-calibration analysis. Engineering-led, but the decision-science instincts are the same.
